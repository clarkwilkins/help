Here’s the core concept: a job is a collection of events (aka “tickets”) performed for one customer (usually at one location). That’s what informs the method now described.

  1. Click dispatch at the top of the screen.
  2. Select the customer account.
  3. If the customer is using locations,1 select as appropriate.
  4. Enter the reason for the job.
  5. Enter the current status.2
  6. Assign the job.3
  7. Add optional notes if desired.4
  8. Dispatch it.

servicd does two things at this point. First, it creates a metadata record for the job to hold things like the customer ( + location?) that won’t change. Second, it creates the first ticket on the job.5 The dichotomy is this: the user on each ticket may change, as well as arrival time, etc, (ticket data), but the job’s customer, job number, etc., will remain the same for all tickets (metadata).

When you open a job for another user, they get a text message immediately letting them know the job was dispatched.

NOTES
  1. Locations are optional.
  2. Status codes define the current status of this job.
  3. Only currently active users are shown.
  4. Notes are private to servicd users. These notes will be sent in text to the assigned user as well.
  5. Tickets show a “ticket number”, but this is just a construct. The truth is that each ticket records when it was “booked”. The first ticket booked is #1, etc.
